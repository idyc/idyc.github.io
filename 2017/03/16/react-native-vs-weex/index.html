React-Native vs Weex

2017-03-16

其实在很早前就准备写一点RN和Weex的对比作为参考。当时Weex还在为支持vue做准备，所以就放置了。在关注了一段时间后，感觉可以做一点简单的比较了。

Playground

RN开发早期的快速预览是在rnplay.org，如今已经下线了，而Weex官方则提供了一个Playground的手机应用，配合weex-toolkit通过扫QR码的方式直接预览，非常方便。不过RN社区的跟进很快，一个新公司Expo上线了作为rnplay替代的snack.expo.io。Facebook官方也与之合作做了CRNP，可见这个公司的工具质量是被官方认可的。现在Expo也提供了手机上的Playground应用，配合XDE不仅可以预览自己的项目，还可以搜索社区中的项目预览，实用度很高。

Navigaton/Router

导航/路由是很重要的基础功能。
首先两者都有基本的Navigator，即常见的导航栏。
但RN需要拿到navigator的引用直接push组件和参数，Weex直接用js文件的url，很不实用。
在实践中，RN有很多开源解决方案，这里以react-native-router-flux(star较多)为例，
Weex则使用官方推荐的vue-router + vuex的组合：

RN有一个值得注意的新方案react navigation，据称是准备统一react和react native的导航方式，不过目前还不稳定，所以暂且不提了。

ListView/TableView

移动应用中一个非常重要的部分是ListView类型的列表组件，不仅是信息展示的重要方式，也对内存提出了较高的要求。

Weex的组件质量比RN低了不少，虽然Weex将很多浏览器的HTML tags移植过来，但并不是完全兼容。在带来方便的同时也容易引起难以预期的bug。template的布局方式对比JSX而言灵活度下降了。

Stability

weex现在成为了一个apache的孵化器项目，不知道这算好消息还是坏消息。不过Weex相关API与其说不稳定，不如说存在混乱；相比较而言RN是演化太快。这里要吐槽一下weex-toolkit和weexpack的割裂，有点意义不明。看了一些相关人员解释说weex-toolkit专注生成jsbundle，而weexpack则用来打包部署。其实这样对于原生开发挺不友好的，不过也有提到将来会将两者合并。

Native develop support

Expo和Facebook合作做了一个create-react-native-app的cli工具，比react-native-cli多集成了expo-sdk。和create-react-app类似的，漂亮的把原生部分藏了起来，让RN项目目录看起来非常干净。需要做原生开发时，可以通过npm run eject重新找到android，ios的原生文件。

Summary

总的来说，Weex比RN简单一些，借鉴了很多RN的内容并有意识的做了简化，但是文档和官方组件的质量较低，坑相对比RN多一些。
在选择开发平台或者工具时，配套设施或者说生态环境，以及开源社区的活跃度都是很重要的，在这些方面Weex落后非常多。
