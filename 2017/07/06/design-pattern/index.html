设计模式的透明化：Akarin’s Perspective

2017-07-06

设计模式是对软件工程中普遍存在，反复出现的问题提出的可复用式的解决方案。这个术语最早在70年代末的建筑设计领域出现，但是却意外的在软件开发领域迅速流行。起因便是GoF（Erich Gamma等4人）编著的《设计模式》一书，共总结了23种设计模式，相应的还有6个设计原则。我们经常能听到关于企业级OO和过度设计的批评，这些先按下不表。理论上来说，设计模式/原则与具体语言和实现无关，而实际上这些设计模式是以C++、Java这些有统治地位的语言为背景的。使用者的思维方式又会受到语言的影响，设计模式的实现方式及简单与否都与之有关系，极端的情况下某些设计模式变成了显而易见的特性：

First-class function

函数式编程语言有一个重要特性叫做First-class function, 意思是指函数是编程语言的第一类公民(first-class citizen)。这意味着函数可以作为别的函数的参数、返回值，赋值给变量或存储在数据结构中。而这个特性会让很多设计模式显得很啰嗦甚至多余。

Pattern Matching

模式匹配是函数式语言中非常强大的功能，在Scala/Kotlin中看起来和Java的switch case很像，但是除了可以对值进行匹配外，还可以对类型匹配，对case class匹配，配合提取器可以匹配任何模式。强大的模式匹配让访问者模式变得没有意义。

Summary

随着现代编程语言越来越多借鉴函数式语言，以往繁杂的设计模式将会消失或透明化，但是其中有一些重要的思想仍然值得借鉴和学习。
